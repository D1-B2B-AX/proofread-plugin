# 교안 검수 플러그인

> 교안(PPT/PDF/노션) 텍스트를 자동 추출하여 오탈자, 맞춤법, 회사명/과정명 불일치, 개인정보 노출 등을 검수하는 플러그인입니다.

## 실행 조건

이 커맨드가 호출되면 아래 절차를 **순서대로** 수행합니다. 각 단계를 건너뛰지 마세요.

---

## STEP 0. 환경 확인 (자동)

플러그인을 최신 버전으로 업데이트합니다. (네트워크 오류 시 무시하고 진행)
```bash
cd ~/.claude/commands/proofread-plugin && git pull --quiet 2>/dev/null || true
```

PPT/PDF 추출에 필요한 패키지가 설치되어 있는지 확인합니다. 없으면 자동 설치합니다.
```bash
pip install python-pptx PyPDF2 2>/dev/null || pip3 install python-pptx PyPDF2 2>/dev/null
```
- 이미 설치되어 있으면 건너뜁니다.
- 결과 저장 폴더가 없으면 생성합니다:
```bash
mkdir -p ~/Desktop/claude_temp
```

## STEP 1. 교안 입력 및 기본 정보 확인

### 1-1. 교안 파일 입력 안내

**파일이 이미 함께 입력된 경우** (슬래시 명령어 + 파일 드래그 동시 입력):
→ 안내 메시지를 건너뛰고 바로 STEP 1-3(기본 정보 확인)으로 진행합니다.
→ **단, STEP 1-4(검수자 이름 확인)는 반드시 거쳐야 합니다. 이름 입력 없이 검수를 시작하지 마세요.**

**파일 없이 명령어만 입력된 경우:**
```
교안 검수를 시작합니다.

교안 파일을 대화창에 드래그해 주세요. (PPT, PDF 지원)
또는 파일 경로를 직접 입력해도 됩니다.

💡 PPT와 PDF 둘 다 있다면 PPT가 더 정확해요.
💡 노션 교안이면 "노션"이라고 입력해 주세요.
💡 수정본 확인은 "2차 검수"라고 말씀해 주세요.
```

### 1-2. 노션 교안인 경우 안내

사용자가 노션 교안이라고 하면, 아래 순서로 안내합니다:

```
노션 교안 입력 방법 (편한 방법을 선택해 주세요):

① 브라우저에서 PDF 저장 (★ 추천)
   → 노션 페이지를 브라우저에서 열기
   → 브라우저 우측 상단 점 3개(⋮) → "인쇄" 클릭
   → 프린터를 "Microsoft Print to PDF"로 변경
   → ⚠️ "머리글 및 바닥글" 체크 해제 (불필요한 URL 제거)
   → "인쇄" 클릭 → 파일 저장
   → 토글 포함 전체 내용이 자동으로 펼쳐져 추출됨

② 텍스트 복사-붙여넣기 (짧은 교안에 한해)
   → 노션 페이지에서 전체 선택(Ctrl+A) → 복사(Ctrl+C) → 여기에 붙여넣기
   → 띄어쓰기 정확도 높음. 단, 토글이 많거나 분량이 긴 교안에는 비실용적

③ 강사에게 마크다운(.md) 내보내기 요청
   → 위 방법이 안 될 때 대안
```

### 1-3. 기본 정보 자동 파악 및 확인

파일이 입력되면, **텍스트 추출 후 앞 5페이지 + 뒤 5페이지를 빠르게 읽어** 고객사명/과정명/강사명을 자동으로 파악합니다.

아래 형식으로 정보를 보여주고 사용자 입력을 기다립니다. 파악하지 못한 항목은 "입력해 주세요"로 표시합니다.

```
파일: [파일명] ([N]p)

아래 정보를 확인하고, 수정할 내용과 검수자 이름을 입력해 주세요:
- 고객사: [파악된 값 또는 "입력해 주세요"]
- 과정명: [파악된 값 또는 "입력해 주세요"]
- 강사명: [파악된 값 또는 "입력해 주세요"]
- 검수자 이름: 입력해 주세요 (메일 서명에 자동 반영)

(예: 롯데백화점 / 전정현)
```

**⚠️ 사용자가 검수자 이름을 포함하여 응답할 때까지 검수를 시작하지 마세요. 파일이 함께 입력된 경우에도 반드시 이름을 받아야 합니다.**

사용자가 이름을 포함하여 응답하면 `review_data`의 `om_name` 필드에 저장하고 검수를 시작합니다. 이름 없이 응답한 경우 "검수자 이름도 함께 입력해 주세요. (예: 전정현)"라고 다시 요청합니다.

## STEP 2. 파일 형식 판별 및 텍스트 추출

- `.pptx` 파일 → 아래 명령 실행:
  ```bash
  PYTHONIOENCODING=utf-8 python3 "~/.claude/commands/proofread-plugin/extract_pptx.py" "파일경로" --json
  ```

- `.pdf` 파일 → 아래 명령 실행:
  ```bash
  PYTHONIOENCODING=utf-8 python3 "~/.claude/commands/proofread-plugin/extract_pdf.py" "파일경로" --json
  ```

- `.md` 파일 (노션 마크다운 내보내기) → Read 도구로 파일 내용을 읽어서 텍스트로 처리

- **텍스트 직접 붙여넣기** (노션 복사-붙여넣기) → 스크립트 실행 없이 붙여넣어진 텍스트를 바로 검수 대상으로 사용. 위치 정보는 제공 불가하므로 위치 컬럼은 "-"로 표시.

- 그 외 확장자 → "지원하지 않는 파일 형식입니다. .pptx, .pdf, .md 파일 또는 텍스트 붙여넣기를 사용해 주세요." 안내 후 종료

추출이 완료되면 사용자에게 간단히 안내:
```
파일명: [파일명]
총 [슬라이드/페이지] 수: [N]p
[이미지 위주 페이지가 있으면] 이미지 위주 페이지: [N]p (텍스트 검수 불가)

텍스트 추출이 완료되었습니다. 검수를 시작합니다.
```

**PDF 교안인 경우 추가 안내:**
```
ℹ️ PDF 교안은 변환 과정에서 띄어쓰기가 누락될 수 있어요.
   명백한 오류만 잡고 애매한 건 넘어가니 안심하세요.
```

## STEP 2.5. 이미지 위주 페이지 OCR (클로드 멀티모달)

PDF 추출 결과에서 `is_image_only: true`인 페이지가 **전체의 30% 이상**일 때만 실행합니다.
(노션→PDF 변환 교안 중 디자인/이미지 중심 교안이 해당)

### 실행 시 안내
```
이미지 위주 페이지가 [N]p ([M]%)로 많아, 이미지 속 텍스트도 함께 확인합니다.
일반 교안보다 시간이 좀 더 소요될 수 있습니다.
```

### 자동 실행 절차

1. **PyMuPDF로 이미지 위주 페이지를 이미지(PNG)로 변환**:
```python
import pymupdf
doc = pymupdf.open(pdf_path)
page = doc[page_index]  # 0-indexed
pix = page.get_pixmap(dpi=150)
pix.save(output_path)
```

2. **변환된 이미지를 Read 도구로 읽어서 클로드가 직접 텍스트 판독**
   - 한 번에 5~10페이지씩 나누어 처리 (토큰 관리)
   - 판독 요청: "이 교안 페이지의 텍스트를 모두 읽어주세요"

3. **판독된 텍스트를 기존 검수 파이프라인에 합류**
   - OCR로 판독된 텍스트에서 발견된 오류는 **"확인 필요"**로 분류 (오류 확정으로 올리지 않음)
   - 위치 정보: "p.[N] (이미지 판독)" 으로 표기

4. **임시 이미지 파일은 검수 완료 후 삭제**

### OCR 운영 원칙
- **best-effort**: 잡히면 bonus, 못 잡아도 정상
- **⚠️ 절대 규칙: OCR 기반 발견 항목은 어떤 경우에도 "오류 확정"으로 분류하지 않는다.** 아무리 명백해 보여도 반드시 **"확인 필요"**로 분류. OCR 오독으로 인한 오탐이 실무를 방해하지 않도록 하기 위함.
- 순수 이미지(일러스트, 사진 등 텍스트가 없는 이미지)는 판독 대상에서 제외
- 노션→PDF 교안은 OCR 오독률이 특히 높으므로(테스트 기준 8건 중 7건 오독), 결과 안내 시 아래 문구를 추가:
  ```
  ⚠️ 이미지 판독(OCR) 기반 항목은 오독 가능성이 있어 모두 "확인 필요"로 분류했습니다. 실제 교안과 반드시 대조해 주세요.
  ```

## STEP 3. 분할 처리 (대용량 교안 대응)

추출된 텍스트를 **약 4,000자 단위**로 나누어 검수합니다.

- 페이지 수가 아닌 **텍스트 양(글자 수)** 기준으로 분할
- 이유: 페이지당 텍스트 양이 0자~500자 이상으로 편차가 크므로, 글자 수 기준이 AI 주의력을 청크마다 균등하게 유지
- 분할 시 페이지 중간에서 끊지 않고, 4,000자에 가장 가까운 **페이지 경계**에서 분할
- 4,000자 이하: 한 번에 검수

### 오버랩 규칙 (문맥 단절 방지)
분할 시 **이전 청크의 마지막 2p를 다음 청크의 앞에 포함**합니다.
- 오버랩 구간에서 이미 보고한 오류는 중복 보고하지 않음
- 오버랩의 목적은 청크 경계에서 놓치는 문맥 오류를 잡기 위함

각 분할 단위마다 STEP 3.5의 요약 컨텍스트와 함께 STEP 4의 검수 기준을 적용합니다.
**분할 처리 시에도 앞쪽 분할에서 발견한 패턴(반복 오류 등)을 뒤쪽 분할에도 동일하게 적용하세요.**

분량이 많을 경우 사전 안내:
```
총 [N]p ([M]자) 분량이므로 [K]회에 나누어 검수합니다. 약 [예상시간]분 소요될 수 있습니다.
```

예상시간 기준:
- 12,000자 이하: 3~5분
- 12,000~30,000자: 5~10분
- 30,000자 이상: 10~20분

## STEP 3.5. 전체 흐름 요약 (문맥 검수 기반 생성)

검수 시작 전, 추출된 전체 텍스트를 빠르게 훑어 아래 요약을 먼저 생성합니다.

**생성할 요약 항목:**
1. **목차/섹션 구조**: 교안의 전체 흐름 (예: 도입→이론→실습→정리)
2. **핵심 용어 목록**: 교안 전반에서 반복 사용되는 주요 용어와 표기 (예: "프롬프트 엔지니어링", "LLM", "RAG")
3. **고객사/과정 맥락**: 교안이 다루는 산업, 대상, 주제의 핵심 맥락

**이 요약은 사용자에게 출력하지 않고**, 이후 각 청크 검수 시 내부 참조용으로 사용합니다.
- 용어 일관성 검수 시: 핵심 용어 목록과 대조
- 문맥 적절성 판단 시: 전체 흐름과 해당 청크의 위치를 고려

## STEP 4. 검수 기준

아래 7가지 기준으로 텍스트를 검수합니다.

### 4-1. 오탈자 검수
- 한글 맞춤법 오류 (예: 됬다→됐다, 어떻게→어떡해 혼동)
- 영문 스펠링 오류 (예: Anlaysis→Analysis)
- 잘못된 조사 사용 (예: "을/를", "이/가", "은/는" 혼동)
- 누락된 글자, 중복된 글자
- **외래어 표기 차이는 오류가 아님**: 실무에서 혼용되는 외래어 표기는 "오류 확정"이 아닌 **"참고사항"**으로 분류. 의미 전달에 문제가 없으므로 강사 수정 요청 불필요.
  - **유사 발음 표기 차이**(ㅔ/ㅐ, ㅗ/ㅓ, ㅜ/ㅡ, ㅅ/ㅆ, ㅋ/ㅎ, ㅂ/ㅍ, ㅈ/ㅊ, ㅌ/ㄷ, 장단음, 받침 유무 등 자음·모음 혼동)도 오류 확정이 아닌 참고사항으로 분류
  - 예: 클레스/클래스, 컨텐츠/콘텐츠, 멤버쉽/멤버십, 메세지/메시지, 유투브/유튜브, 세그멘트/세그먼트, 플렛폼/플랫폼 등

### 4-1b. 의심 단어 검수 (형태소 분석 자동 감지)
- 추출 스크립트가 `suspicious_words` 필드에 **사전에 없는 한글 단어**를 자동 감지하여 제공
- 형태소 분석기(kiwipiepy)가 단어를 낱글자로 쪼개는 경우 = 사전에 없는 단어 의심
- **반드시 이 데이터를 확인하고 검수 결과에 반영하세요** — 클로드가 놓치기 쉬운 오탈자를 잡아주는 핵심 도구
- 예시: "인프로" → ["인", "프로"]로 분리됨 → "인프라"의 오탈자 의심
- 각 의심 단어에 대해 문맥을 확인하여:
  - 명백한 오탈자 → "오류 확정" (예: 인프로→인프라, 섬요→섬유)
  - 정상 단어인데 형태소 분석기가 오분석 → 무시 (예: 세대, 전세계, 그라데이션 등 흔한 외래어나 합성어)
  - 판단 어려움 → "확인 필요"

### 4-1c. 영문 스펠링 검수 (자동 감지)
- 추출 스크립트가 `suspicious_english` 필드에 **영문 사전에 없는 단어**를 자동 감지하여 제공
- 대문자 약어(SQL, DBMS 등), 코드, 파일 경로, camelCase는 자동 필터링됨
- 각 의심 단어에 대해 문맥을 확인하여:
  - 명백한 스펠링 오류 → "오류 확정" (예: Linve→Line, Anlaysis→Analysis)
  - 고유명사/브랜드명 → 무시 (예: Canva, Unsplash, Worksmart 등)
  - 판단 어려움 → "확인 필요"

### 4-1d. 반복 텍스트 변형 검수 (자동 감지)
- 추출 스크립트가 `suspected_variants` 필드에 반복 텍스트의 소수 변형을 자동 감지하여 제공
- **반드시 이 데이터를 확인하고 검수 결과에 반영하세요** — 문맥상 정상인 단어도 빈도 비교로 잡아낼 수 있음
- 예시: "전환"이 3회, "전한"이 1회 → "전한"을 변형 의심으로 플래그
- 각 변형에 대해 문맥을 확인하여:
  - 명백한 오탈자 → "오류 확정"
  - 의도적 차이 가능 (연구소 A vs B 등) → 무시
  - 판단 어려움 → "확인 필요"

### 4-1e. 텍스트 깨짐 검수 (자동 감지)
- 추출 스크립트가 `broken_texts` 필드에 **글자 깨짐이 감지된 텍스트**를 자동 제공
- 폰트 임베딩 실패, 인코딩 불일치 등으로 글자가 정상 표시되지 않는 경우를 감지
- 감지 유형:
  - **대체 문자(□)**: 폰트가 없는 환경에서 PDF 변환 시 한글이 □로 치환 (예: `2024□1 □□□□□`)
  - **유니코드 대체 문자(�)**: 인코딩 변환 실패로 글자가 �로 표시 (예: `2024�1 �����`)
  - **자모 분리**: 완성형 한글이 초성/중성/종성으로 풀어져 표시 (예: `ㅌㅗㅇ이 ㄹ` ← "통일")
- 모든 깨짐 감지 항목은 **"오류 확정"**으로 분류 (깨짐은 누가 봐도 오류)
- 심각도는 **"상"** (고객 눈에 바로 보이는 오류)

### 4-2. 띄어쓰기 검수 (자동 감지 + LLM 판단)
- 추출 스크립트가 `suspected_spacing` 필드에 **띄어쓰기 오류 후보**를 자동 감지하여 제공
- kiwi.space() 형태소 분석 기반으로 의존명사 붙어쓰기, 보조용언 붙어쓰기 등을 포괄적으로 감지
- **반드시 이 데이터를 확인하고 검수 결과에 반영하세요** — LLM만으로는 회차마다 누락이 발생할 수 있음
- 각 후보에 대해 문맥을 확인하여:
  - 명백한 띄어쓰기 오류 → "오류 확정" (예: "생성시"→"생성 시", "할수있다"→"할 수 있다")
  - 교안 특성상 의도적 표기 가능 → "확인 필요"
  - 전문 용어·합성어의 과도한 분리 (오탐) → 무시 (예: "머신러닝"→"머신 러닝")
- **PDF에서 추출된 텍스트의 경우**: 원본 PDF 특성상 띄어쓰기가 누락될 수 있음. 명백한 경우만 오류로 잡고, 애매한 경우는 "확인 필요"로 분류
- **⚠️ "~화 하다/된" 패턴은 PDF 추출 문제로 넘기지 말 것**: "최적화 한다", "체계화 된", "고려 하여", "개인화 하여" 등 **용언 파생 접미사 뒤의 불필요한 띄어쓰기**는 PDF 추출 한계가 아니라 실제 띄어쓰기 오류일 가능성이 높음. suspected_spacing에서 감지된 이런 패턴은 "오류 확정" 또는 최소 "확인 필요"로 반영할 것

### 4-3. 전각문자 검수
- 전각 괄호, 전각 숫자 등 의도치 않은 전각문자 사용 확인
- 단, 디자인 의도로 사용된 경우도 있으므로 "확인 필요"로 분류

### 4-4. 고객사명/과정명 일치 확인
- 사용자가 입력한 **고객사명**과 교안 내 표기가 일치하는지 확인
- **표기 변형은 오류가 아님**: KT / 케이티 / kt 는 모두 같은 회사이므로 오류로 잡지 않음
- **다른 기업명 발견 시**: "오류 확정"이 아닌 **"확인 필요"**로 분류 (사례/케이스스터디/계열사일 수 있음)
- **과정명**도 동일 기준 적용: 입력한 과정명과 다른 과정명이 나오면 "확인 필요"

### 4-5. 개인정보/민감정보 노출 검수
- 전화번호 패턴 (010-XXXX-XXXX 등) → **"확인 필요"** (강사 연락처 의도 가능)
- 이메일 주소 → **"확인 필요"** (강사 연락처 의도 가능)
- 주민등록번호 패턴 → **"오류 확정 / 상"** (의도적 노출일 수 없음)
- 기타 민감 정보 (계좌번호, 카드번호 등) → **"오류 확정 / 상"**
- **분류 고정 규칙**: 위 기준은 매 검수마다 동일하게 적용. LLM 재량으로 변경하지 않음

### 4-6. 강사명 확인
- 사용자가 입력한 **강사명**이 교안 내에 정상적으로 표기되어 있는지 확인
- 다른 강사명이 발견되면 "확인 필요"로 분류 (이전 교안 재활용 가능성)

### 4-7. 문맥 기반 검수 (문장/문단 수준)

STEP 3.5에서 생성한 전체 흐름 요약을 참조하여 아래를 검수합니다.

#### 주술 호응
- 주어와 서술어가 호응하지 않는 문장 (예: "이 기능은 ~할 수 있다" → "이 기능을 사용하면 ~할 수 있다")
- 주어가 생략된 채 서술어만 있는 경우는 교안 특성상 정상이므로 오류로 잡지 않음

#### 용어 일관성
- 같은 개념을 가리키는데 표기가 섞인 경우 (예: "프롬프트" vs "Prompt"가 혼용, "워크플로우" vs "워크플로")
- STEP 3.5의 핵심 용어 목록과 대조하여 불일치 감지
- 의도적 혼용일 수 있으므로 **"확인 필요"**로 분류 (오류 확정 아님)

#### 문장 완결성
- 문장이 중간에 끊긴 경우 (예: "이를 통해 우리는", "다음으로 살펴볼 내용은")
- 접속사로 시작하지만 이어지는 내용이 없는 경우
- 단, 슬라이드 제목이나 불릿 항목의 체언 종결(명사로 끝남)은 교안 특성상 정상

#### 같은 슬라이드 내 제목/코드 일치
- 슬라이드 제목(또는 소제목)에 기법명·키워드가 있고, 같은 슬라이드에 코드가 있을 때, **제목이 말하는 내용과 코드가 일치하는지** 반드시 확인
- 예: 제목 "Inner Join - 추가조건"인데 코드에 `LEFT JOIN`이 사용됨 → 불일치
- 코드 교육 교안에서 흔히 발생하는 오류 (이전 슬라이드 코드를 복사하면서 제목만 바꾸거나, 제목을 안 바꾸는 경우)
- **"오류 확정"**으로 분류

#### 순서/단계 패턴 대조
- 교안 내에서 **정해진 순서가 있는 용어 체계**(단계명, 프레임워크 등)가 사용될 때, 해당 순서가 맞는지 반드시 대조
- 예: 퍼널 모델 TOFU(Top) → MOFU(Middle) → BOFU(Bottom) 순서에서, BOFU가 와야 할 자리에 TOFU가 적혀있으면 오류
- 예: "1단계 → 2단계 → 4단계" (3단계 누락 또는 번호 오기)
- STEP 3.5의 전체 흐름 요약에서 파악한 프레임워크/단계 체계를 기준으로 대조
- **"확인 필요"**로 분류


#### 앞뒤 문맥 모순
- 앞 페이지에서 설명한 내용과 뒤 페이지 내용이 모순되는 경우
- 예: p.5 "3단계로 진행" → p.8 "4단계 프로세스" (단계 수 불일치)
- 오버랩 구간 및 전체 흐름 요약을 활용하여 감지
- **"확인 필요"**로 분류

#### 유사 단어 문맥 대조
- **사전에 존재하지만 문맥에 안 맞는 단어**를 잡는 기준
- 2글자 한글 단어 중 1글자만 바뀌면 완전히 다른 뜻이 되는 경우, 주변 문맥과 대조하여 원래 의도된 단어가 맞는지 확인
- **판단 방법:**
  - 같은 페이지 내 다른 항목과 패턴 대조 (예: "서비스강화", "생산성향상" 옆에 "보안 강력" → "강화"가 맞을 가능성)
  - 문장 전체의 의미 흐름에서 해당 단어가 자연스러운지 확인 (예: "배움이 있는 교생, 변화하는 학교" → "교실"이 더 자연스러움)
  - STEP 3.5의 전체 흐름 요약과 대조 (교안 주제와 동떨어진 단어가 갑자기 등장하면 의심)
- 확신이 있으면 "오류 확정", 아니면 **"확인 필요"**로 분류

## STEP 5. 검수 중 셀프 체크 (내장형 검증)

각 분할 단위 검수가 끝날 때마다 아래를 자체 확인:

1. **오탐 점검**: "이 오류가 실제로 맞는가?" — 전문 용어, 고유명사, 의도적 표현은 오류가 아님
2. **누락 점검**: "놓친 오류가 없는가?" — 텍스트를 다시 한번 훑어보기
3. **분류 점검**: "오류 확정 vs 확인 필요 분류가 맞는가?" — 확실하지 않으면 "확인 필요"로
4. **OCR 분류 점검**: OCR(이미지 판독) 기반 항목이 "오류 확정"에 들어가 있지 않은가? → **OCR 항목은 전부 "확인 필요"로 이동**

**모든 분할 검수가 완료된 후**, 결과 출력 전에 아래 **기계적 대조 검증**을 반드시 수행:

4. **suspicious_words 전수 대조** (Verify 축):
   - 추출 스크립트가 감지한 `suspicious_words` 목록의 **모든 항목**을 하나씩 확인
   - 각 의심 단어에 대해: ① 검수 결과에 이미 포함했는가? ② 안 했다면 해당 페이지 문맥을 재확인
   - 정상 단어로 판단되면 무시, 오류로 판단되면 검수 결과에 추가
   - **이 단계를 건너뛰지 마세요** — 형태소 분석기가 잡아낸 후보를 클로드가 놓치면 감지 기능의 의미가 없음

5. **suspected_variants 전수 대조** (Verify 축):
   - `suspected_variants` 목록의 모든 항목도 동일하게 대조
   - 검수 결과에서 다루지 않은 변형이 있으면 문맥 재확인 후 판단

6. **broken_texts 전수 반영** (Verify 축):
   - `broken_texts` 목록의 모든 항목을 검수 결과에 **"오류 확정 / 심각도 상"**으로 포함
   - 깨짐은 판단이 필요 없는 명백한 오류이므로 누락 없이 전부 반영

7. **suspected_spacing 전수 대조** (Verify 축):
   - `suspected_spacing` 목록의 모든 항목을 하나씩 확인
   - 명백한 띄어쓰기 오류 → 검수 결과에 추가
   - 전문 용어·합성어 오탐 → 무시
   - **이 단계를 건너뛰지 마세요** — LLM만으로는 띄어쓰기 검출이 회차마다 누락될 수 있음

**원칙: 확실한 오류만 "오류 확정", 조금이라도 의도적일 수 있으면 "확인 필요"**

## STEP 6. 결과 출력 (HTML)

검수 결과를 **HTML 파일로 생성**하고 브라우저에서 자동으로 열어 보여줍니다.

**HTML 생성 방법:**
검수 결과 데이터를 아래 스크립트에 전달하여 HTML을 생성합니다:
```python
import sys; sys.path.insert(0, r'~/.claude/commands/proofread-plugin')
from generate_html import generate_html, open_in_browser

review_data = {
    "file_name": 파일명,
    "client": 고객사명,
    "course": 과정명,
    "instructor": 강사명,
    "om_name": 검수자이름,
    "file_type": "ppt" | "pdf" | "notion" | "text",
    "errors": [...],      # 오류 확정 목록
    "warnings": [...],    # 확인 필요 목록
    "notes": [...],       # 참고사항 목록
    "mail_subject": 메일 제목
}
path = generate_html(review_data)
open_in_browser(path)
```

각 항목의 필드:
- errors: `{"page", "type", "severity"(상/중/하), "original", "suggestion", "location"}`
- warnings: `{"page", "type", "original", "suggestion", "location"}`
- notes: `{"page", "type", "original", "suggestion", "note"}`

### 6-1. HTML 구성

**상단 헤더 (Day1 블루 #13299f 배경, 흰색 글씨):**
- 타이틀: 「[과정명]」| 교안 검수 결과
- 부제: [고객사명] · [강사명] 강사 · [검수일]
- 기본 정보 박스: 파일명, 고객사, 과정명, 강사명
- 요약 카드 3개: 오류 확정(빨강) / 확인 필요(주황 #e67700) / 참고(회색)

**하단 상세 (흰색 배경):**
- 안내 문구 (교안 형태별 자동 변경):
  - PPT: "페이지 번호는 PPT 왼쪽 슬라이드 목록 순서와 동일합니다."
  - PDF: "페이지 번호는 PDF 뷰어 페이지 번호와 동일합니다. PDF 특성상 띄어쓰기가 누락될 수 있으며, 이는 원본 오류가 아닙니다."
  - 노션/텍스트: "텍스트 직접 입력이므로 위치 정보는 제공되지 않습니다."
- 오류 확정 테이블 (행 좌측 빨간 바, 심각도 뱃지, 페이지 번호 파란색)
- 확인 필요 테이블 (행 좌측 주황 바)
- 참고사항 테이블 (행 좌측 회색 바)
- 메일 초안 (체크박스로 항목 선택 → "메일 복사" 버튼 1개)

### 6-2. 오류 0건 처리

오류 확정이 0건일 경우:
- 오류 확정 테이블 대신 "오류가 발견되지 않았습니다" 메시지 표시
- 메일 초안 섹션 숨김 (강사에게 보낼 내용 없음)
- 확인 필요, 참고사항은 그대로 표시

### 6-3. 분류 기준

- **오류 확정**: 누가 봐도 틀린 것 (오탈자, 잘못된 조사, 개인정보 노출, 다른 강사명 잔존 등)
- **확인 필요**: 의도일 수 있는 것 (타사명 사례, 날짜 형식 차이, 문법적으로 애매한 표현 등)
  - **확인 필요 심각도 "상"**: 교안 내용의 정확성 확인이 필요한 항목. OM이 직접 판단하기 어려워 강사에게 확인이 필요한 경우. **메일에 포함됨.** 해당 유형:
    - 코드/주석 불일치 (주석 설명과 실제 코드가 다른 경우)
    - 코드 자체 오류 (SQL 문법, 키워드 오류 등)
    - 수식/공식 불일치 (엑셀 수식 설명과 실제 수식이 다른 경우)
    - 프롬프트 예시 불일치 (프롬프트 설명과 실제 프롬프트 내용이 다른 경우)
  - **확인 필요 심각도 없음(-)**: 띄어쓰기, 개인정보, OCR 항목 등 OM이 직접 판단 가능한 항목. **메일에 미포함.**
- **참고사항**: 틀리진 않지만 알아두면 좋은 것 (외래어 표기, 표기 통일 권장 등)
- **⚠️ 분류 고정 유형** — 아래 유형은 매 검수마다 반드시 동일하게 분류 (LLM 재량 금지):
  - 코드/주석 불일치 → 확인 필요 / 상
  - 코드 자체 오류 → 확인 필요 / 상
  - 수식/공식 불일치 → 확인 필요 / 상
  - 프롬프트 예시 불일치 → 확인 필요 / 상
  - 이메일·전화번호 → 확인 필요
  - 주민등록번호·계좌번호 → 오류 확정 / 상
  - OCR 기반 발견 항목 → 확인 필요
  - 외래어 표기 차이 → 참고사항
  - 다른 강사명 잔존 → 확인 필요
  - 다른 고객사명/과정명 발견 → 확인 필요
- **심각도** (오류 확정): 상(고객 눈에 바로 보이는 오류) / 중(주의 깊게 보면 발견) / 하(사소한 표기)
- **위치**: [상] 페이지 상단 / [중] 페이지 중간 / [하] 페이지 하단 — 해당 문맥 일부 포함
- **정렬 기준**: 오류 확정·확인 필요·참고사항 모두 **페이지 번호 오름차순**으로 정렬 (OM이 교안을 앞에서 뒤로 넘기며 확인하기 편하도록). 심각도는 뱃지 색상으로 시각 구분

### 6-4. 파일 저장

- HTML 저장 경로: `~/Desktop/claude_temp/검수결과_[파일명]_[날짜].html` (자동 생성)
- 생성 후 `webbrowser.open()`으로 브라우저 자동 열기
- 대화창에는 아래와 같이 요약 + 해석 안내를 출력:

검수 완료. 브라우저에서 상세 결과를 확인해 주세요.

📌 오류 확정 X건 — 강사에게 수정 요청이 필요한 항목
📌 확인 필요 Y건 — 의도일 수 있어 직접 판단이 필요한 항목
📌 참고 Z건 — 틀리진 않지만 알아두면 좋은 항목 (강사 전달 불필요)

## STEP 7. 강사 수정 요청 메일 초안

**[오류 확정]**이 1건 이상 또는 **[확인 필요(상)]**이 1건 이상 발견되면 결과 페이지에 메일 초안이 자동 생성됩니다.

**v2.1 체크박스 기반 메일 구성:**
- 결과 페이지에서 오류 확정 + 확인 필요(상) 항목에 **체크박스**가 표시됨
- OM이 체크박스로 강사에게 요청할 항목을 선택하면 **메일 본문이 자동 재구성**됨
- "메일 복사" 버튼 클릭 → 제목+본문이 클립보드에 복사됨
- 발신자명은 STEP 1-4에서 입력한 `om_name`이 자동 반영됨 ("검수자이름 드림")

**메일 톤 기준:**
- "수정 요청"이 아닌 **"확인 요청"** 톤
- 강사에 대한 배려 표현 포함
- **[확인 필요(중/하 및 심각도 없음)] 항목은 체크박스 없음** (OM이 직접 판단 후 필요 시 별도 전달)
- **[참고사항] 항목은 체크박스 없음** (OM 내부 참고용이므로 강사 전달 불필요)

**메일 제목 (review_data의 mail_subject에 저장):**
```
[고객사명]_[과정명] 교안 확인 요청드립니다
```

**메일 본문은 JavaScript에서 체크된 항목 기반으로 동적 생성됩니다.** Claude가 mail_body를 직접 생성할 필요 없이, `mail_subject`만 review_data에 포함하면 됩니다.

**메일 복사 기능:**
HTML 결과 페이지에 "메일 복사" 버튼 1개가 포함되어 있으므로, 별도 클립보드 복사 명령은 불필요합니다.

## STEP 8. 2차 검수 (수정본 반영 확인)

사용자가 "2차 검수"를 요청하면:

1. 1차 검수 결과 파일(`검수결과_*.html`)을 불러옴
2. 수정된 교안 파일 경로를 입력받음
3. 수정본에서 텍스트 재추출
4. 1차에서 발견된 오류 항목 각각에 대해:
   - **수정 완료**: 해당 오류가 수정안대로 반영됨
   - **미수정**: 해당 오류가 그대로 남아있음
   - **다르게 수정**: 수정안과 다른 방식으로 변경됨 (확인 필요)
5. 결과를 테이블로 출력:

| # | 페이지 | 오류 유형 | 원문 | 수정 제안 | 반영 상태 |
|---|--------|----------|------|----------|----------|
| 1 | p.3 | 오탈자 | "됬습니다" | "됐습니다" | 수정 완료 |
| 2 | p.15 | 띄어쓰기 | "할수있다" | "할 수 있다" | 미수정 |

6. 새로운 오류가 추가 발견되면 별도 테이블로 표시
7. 수정 반영 요약: "총 X건 중 Y건 수정 완료 / Z건 미수정"

## 주의사항

1. **PDF 띄어쓰기 복원**: PDF에서 추출된 텍스트는 원본 특성상 띄어쓰기가 누락될 수 있습니다. "서로다른사람들이" 같은 패턴이 보이면, 이는 PDF 추출 과정의 한계일 가능성이 높습니다. 명백한 맞춤법 오류만 잡고, PDF 추출 한계로 보이는 것은 제외하세요.

2. **이미지 위주 페이지**: `is_image_only: true`인 페이지는 검수 대상에서 제외하고, 결과에 "이미지 위주 페이지 [N]p는 텍스트 검수가 불가합니다"로 안내합니다.

3. **전문 용어/고유명사**: IT, AI, LLM, API 등 전문 약어나 고유명사(브랜드명, 제품명 등)는 오류로 잡지 마세요.

4. **코드/URL**: 프로그래밍 코드 블록, URL, 이메일 주소(강사 연락처로 의도된 것) 등은 오탈자 검수 대상에서 제외하세요.

5. **표 데이터**: `[표]`로 시작하는 줄은 표에서 추출된 텍스트입니다. 표 내용도 동일한 검수 기준을 적용하되, 표 특성상 축약 표현이 많을 수 있으므로 유의하세요.

6. **결과 저장은 항상**: 검수 결과는 HTML 파일로 저장하고 브라우저에서 열어 보여주세요.
